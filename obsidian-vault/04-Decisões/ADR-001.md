# ADR-001: Stack Tecnológico

Espelho de `docs/decisions/0001-stack-tecnologico.md` (fonte de verdade — atualize lá primeiro).

**Status:** Aceito · **Data:** 2026-05-26 · **Decisores:** Diego

## Contexto

Solo developer, prazo de 2-3 meses para o MVP, MCPs já configurados localmente (`context7`,
`Firebase`, `n8n-mcp`, `render`, `shadcn`, `Stitch`, `supabase-mcp-server`, `upstash`),
preferência declarada por stack "tudo integrado via MCP". Proposta original (React+TS+NestJS/
Python+Postgres+Cloud) não amarrava a MCPs.

## Decisão

Stack inteiramente baseado nos MCPs já configurados — ver [[Arquitetura Geral]] para a versão
atual e completa (React 18 + Vite + shadcn/ui + Supabase + QStash + Redis + OpenRouter).

## Alternativas consideradas

- NestJS/FastAPI + infra dedicada — rejeitada, viola a preferência de "tudo via MCP"
- Firebase em vez de Supabase — rejeitada, domínio fortemente relacional (Firestore é NoSQL)
- n8n para o pipeline — reservado para automações tangenciais, não o pipeline principal

## Consequências

**Boas:** setup quase zero; custo estimado $3–28/mês; TypeScript fim-a-fim; Realtime do
Supabase resolve a UI durante processamento assíncrono.

**Tradeoffs:** dependência de 3 serviços externos; timeout de 150s nas Edge Functions (resolvido
por [[Backend|QStash]]); free tier vai exigir upgrade; chamadas de IA isoladas atrás de camada
fina para facilitar troca de provedor.

**Como reverter:** schema é Postgres puro (migrável); frontend é React puro (migrável); QStash
substituível por Inngest/Trigger.dev/fila no Postgres.

Ver [[Frontend]], [[Backend]], [[Supabase]].
